Transparent O*NET to NOC 2021 Mapping

Data version

This document uses mapping version: 2.0.0

1. Why not just use OaSIS?

OaSIS provides occupation-level skill profiles for Canadian classifications, making it an attractive alternative. However, it does not expose how those profiles are constructed.

  • No observable mapping between O*NET and NOC occupations
  • No weights linking underlying occupations
  • No information on how multiple occupations are combined

As a result, users cannot audit or replicate how each skill profile is formed.

This project instead constructs the mapping explicitly, allowing:

  • full transparency of inputs and weights
  • diagnostic evaluation of mapping quality
  • reproducibility of all outputs

2. Why naive crosswalk chaining fails

A common approach is to join crosswalks:

O*NET → SOC → NOC

This produces many-to-many expansions and implicitly assigns equal weight across all paths.

This leads to:

  • over-weighting of occupations that appear frequently in crosswalk chains
  • inclusion of unrelated occupations
  • distorted skill vectors

For instance, consider 72600: Air pilots, flight engineers and flying instructors. The naive mapping would apply equal weighting to the following rows:

Code 
naive_mapping|> 
  filter(noc_plus_title == "72600: Air pilots, flight engineers and flying instructors")|>
  select(onet_plus_title, noc_plus_title)|>
  reactable(
  columns = list(onet_plus_title = colDef(name = "O*NET occupation"),
                 noc_plus_title = colDef(name = "NOC")),
  searchable = FALSE,
  filterable = FALSE,
  sortable = TRUE,
  pagination = FALSE,
  bordered = TRUE,
  striped = TRUE,
  highlight = TRUE
)

Only \(\frac38\) of the skill profile for NOC 72600: Air pilots, flight engineers and flying instructors is coming from pilots! The other \(\frac58\) is derived from the skill profiles of agricultural workers.

3. Mapping construction

The mapping is constructed in three stages:

  1. O*NET 2019 → SOC 2018
  2. SOC 2018 → NOC 2016
  3. NOC 2016 → NOC 2021

Each stage introduces potential one-to-many relationships.

Path weights

Weights are assigned as:

\[ w_{path} = w_1 \times w_2 \times w_3 \]

where each stage distributes mass equally across branches.

Aggregation

Weights are aggregated to the NOC level and normalized.

Downweighting

A power transformation is applied:

\[ \tilde{w}_k \propto w_k^{2} \]

to reduce the influence of small-weight links.

Now we can return to our 72600: Air pilots, flight engineers and flying instructors example, and investigate these weights.

Code 
mapping|> 
  filter(noc_plus_title == "72600: Air pilots, flight engineers and flying instructors")|>
  select(onet_plus_title, noc_plus_title, base_weight, down_weight)|>
  reactable(
  columns = list(onet_plus_title = colDef(name = "O*NET occupation"),
                 noc_plus_title = colDef(name = "NOC"),
                 base_weight= colDef(name = "Base Weight", format = colFormat(digits = 2)),
                 down_weight= colDef(name = "Down Weight", format = colFormat(digits = 2))
                 ),
  searchable = FALSE,
  filterable = FALSE,
  sortable = TRUE,
  pagination = FALSE,
  bordered = TRUE,
  striped = TRUE,
  highlight = TRUE
)

Using the base weights, 78% of the skill profile is being derived from pilots. Recall in the naive case, only 37.5% of the skill profile came from pilots. Things improve further if one uses the down weight: now 96% of the skill profile is derived from pilots. This is the most extreme example of a difference in skill profile between the naive weights and either the base or down weights.

5. Diagnostics for interpretation

Two key diagnostics are provided:

Mapping concentration (Herfindahl)

\[ H = \sum_k w_k^2 \]

Interpretation:

  • High H → well-defined occupation
  • Low H → heterogeneous occupation

Sensitivity to naive mapping (Δ)

\[ \Delta = || s_{naive} - s_{weighted} || \]

Interpretation:

  • Large Δ → naive mapping is misleading
  • Small Δ → naive mapping is acceptable

6. Interpreting H and Δ

These measure different things:

  • H: whether a single skill profile is meaningful
  • Δ: whether naive mapping introduces error

Cases

  • High H, low Δ → stable and well-defined
  • High H, high Δ → naive mapping contaminated
  • Low H, NOC is inherently too broad, no way to fix

8. Guidance for use

  • Use down weight mapping as default
  • Treat mapped skill profiles of low-H occupations with caution
  • Avoid naive mapping

9. Reproducibility
